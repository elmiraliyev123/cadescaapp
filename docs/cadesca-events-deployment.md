# Cadesca Events deployment checklist

Deploy application code and the Events database migration as one coordinated release. The UI is defensive when Events data is unavailable, but onboarding, ticketing, finance, and scanning require the migration.

## Supabase

1. Review and apply `supabase/migrations/20260718081640_cadesca_events.sql` in a staging branch first.
2. Run the Supabase security and performance advisors after the migration and resolve every Events-related finding.
3. Confirm the `event-assets`, `club-verification`, and `event-receipts` buckets exist with the migration-defined privacy and MIME/size limits.
4. Keep `SUPABASE_SERVICE_ROLE_KEY` server-only. Do not expose it through a `NEXT_PUBLIC_` variable.
5. Configure Resend with `RESEND_API_KEY` and a verified `EMAIL_FROM` sender for representative OTP messages. Supabase SMTP is only needed for mail generated by hosted Supabase Auth itself.
6. Enable Auth leaked-password protection before opening representative registration.
7. Verify Auth redirect URLs include the canonical auth and student-club origins.
8. Confirm the new tables are not unintentionally exposed through Data API grants. Application mutations run through server-authorized routes/private database functions.
9. Confirm `private` is not listed as an exposed Data API schema in the hosted project.

`supabase/config.toml` configures local development only; do not copy it as a production security baseline. In particular, configure hosted password rules, email confirmation, SSL/network restrictions, and Auth URLs independently in the Supabase dashboard.

The linked project does not currently have `pg_cron`; reservation cleanup is therefore triggered by Vercel Cron. Every capacity-sensitive database function also expires stale reservations before making a decision, so the cron is cleanup rather than the sole source of correctness.

## Vercel environment

Required existing values remain unchanged. Add:

- `NEXT_PUBLIC_STUDENT_CLUB_ORIGIN=https://studentclub.cadesca.com`
- `CRON_SECRET` — a high-entropy server-only value used by Vercel Cron
- `NEXT_PUBLIC_COOKIE_DOMAIN=.cadesca.com`

Also confirm these existing server-only values are present:

- `AUTH_SECRET`
- `DATABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `RESEND_API_KEY`
- `EMAIL_FROM`
- `TURNSTILE_SECRET_KEY`

And confirm these intentionally browser-visible project values are present:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY`
- `NEXT_PUBLIC_TURNSTILE_SITE_KEY`

Register `studentclub.cadesca.com` as an allowed hostname for the Cloudflare Turnstile widget and verify that its public and secret keys are a matching pair. Keep `TURNSTILE_ALLOWED_HOSTNAMES=studentclub.cadesca.com` in production.

For event and club image moderation, keep the existing Cloudflare moderation credentials configured and set `CLOUDFLARE_IMAGE_MODERATION_REQUIRED=true`. Event covers and club logos fail closed in production even if this flag is omitted.

## DNS and domains

1. Add `studentclub.cadesca.com` to the same Vercel project as the current Cadesca deployment.
2. Create the DNS record Vercel requests and wait for TLS issuance.
3. Keep representative onboarding routes on `studentclub.cadesca.com` only.
4. Keep all future representative login on `auth.cadesca.com/login`.
5. Keep discovery, tickets, club tools, finance, and scanning on `app.cadesca.com`.
6. Keep public-safe event canonicals on `cadesca.com/event/{slug}`.
7. Confirm the shared `.cadesca.com` cookie configuration works across auth, app, and student-club domains.

Use the exact CNAME value shown by the project domain inspector instead of assuming a generic target; Vercel documents the current subdomain flow in [Adding & Configuring a Custom Domain](https://vercel.com/docs/domains/working-with-domains/add-a-domain).

## Vercel Cron

`vercel.json` calls `/api/cron/events/expire` every five minutes. Vercel sends `Authorization: Bearer <CRON_SECRET>` automatically when the project secret is configured. Verify one successful invocation in deployment logs before launch.

Confirm the Vercel team plan supports a five-minute schedule before deploying it. Vercel Hobby projects are currently limited to once-daily cron jobs; use a supported paid plan or an equivalent authenticated scheduler for this cadence. See [Vercel's cron management documentation](https://vercel.com/docs/cron-jobs/manage-cron-jobs).

Cron delivery is best-effort. Alert on failed cleanup invocations and document a manual reconciliation run that calls the same endpoint with `Authorization: Bearer <CRON_SECRET>`. Cleanup is idempotent and every capacity-sensitive operation also expires stale reservations, so a delayed cron cannot oversell capacity.

## Launch verification

Run the repository checks, apply the migration to staging, and exercise:

- representative OTP, application, clarification, approval, and future auth login;
- approved-club event draft, moderation, publish, and feature controls;
- final-slot concurrent reservations and expiration;
- bank receipt privacy, cash deadlines, free ticket modes, and finance review;
- Cadesca QR preview, explicit confirmation, and simultaneous duplicate check-in;
- EN, AZ, TR, and RU at 320, 375, 390, 430, 1280, and 1440 pixels.
